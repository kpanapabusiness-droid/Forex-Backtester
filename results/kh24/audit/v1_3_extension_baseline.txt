Note on serialisation parity: the legacy-subset sha256 is NOT identical
to the full-file sha256 above, because slicing-then-reserialising goes
through pandas' to_csv (vs the original generator's to_csv on the full
DataFrame). What matters is that the slice-then-reserialise pipeline
is deterministic: running it on the pre-change file gives one sha; the
post-change file must produce the same sha. The baseline_legacy_sha
below was populated by the helper script
scripts/_v1_3_record_legacy_sha.py which performs exactly the same
slice-and-hash the regression test does.

BASELINE_FULL_FILE_SHA:      d5f99d5692ca57fbbbab004fbf95f51206e0d8b3266f699f737158d9bf72fd9e
BASELINE_LEGACY_SUBSET_SHA:  29a056b7932a4019627edd23b08913c9f2459a8160618e869e7471526be230d2
  (slice-and-reserialise with explicit lineterminator='\n' for cross-
   platform stability — pandas defaults to os.linesep which yields CRLF
   on Windows and LF on Linux, producing different hashes for the same
   underlying values. On Windows dev with default settings the same
   slice produces sha 55d617da6009dfb2548feabc6bc9573e749a83c564a9463c78f50ab05c1e101e;
   both encode the same logical data, but only the LF-normalised one
   is platform-stable. CI runs on Linux so the LF sha is the canonical
   regression baseline.)
